# Strava Dashboard

Personal running analytics platform built on top of the Strava API. Collects activity data, computes training load metrics (ACWR, monotony, strain), and provides a unified view of running performance over time.

> **Status:** Active development — backend and data sync layer complete, analytical engine and frontend in progress.

## Overview

The platform syncs running activities from Strava, stores them in a PostgreSQL database, and exposes them for analytical queries. The longer-term vision is a unified fitness hub aggregating multiple data sources (Strava, Hevy, diet tracking) to enable cross-domain correlation.

Key capabilities:

- OAuth 2.0 integration with Strava (token exchange, automatic refresh)
- Incremental activity sync via scheduled cron job and on-demand HTTP endpoint
- Per-activity splits, lap detection, and second-by-second stream storage
- Workout classification (easy/long, interval, hill repeats) based on activity description
- Training load metrics computed via PostgreSQL window functions

## Getting Started

### Prerequisites

- Node.js 20+ (via [nvm](https://github.com/nvm-sh/nvm))
- Docker (for PostgreSQL)
- A [Strava API application](https://www.strava.com/settings/api)

### Environment Variables

Create a `.env` file inside `server/`:

```env
# Database
DATABASE_URL=postgresql://user:password@localhost:5432/strava_dashboard

# Strava OAuth
STRAVA_CLIENT_ID=your_client_id
STRAVA_CLIENT_SECRET=your_client_secret
STRAVA_REDIRECT_URI=https://localhost

# Single-user seed (cuid generated by `npx prisma db seed`)
SEED_USER_ID=your_seed_user_id
```

### Running the Server

```bash
# Start PostgreSQL
docker run --name strava-pg -e POSTGRES_PASSWORD=password -p 5432:5432 -d postgres:17

# Install dependencies
cd server
npm install

# Run migrations
npx prisma migrate dev

# Seed the database (creates the single user + StravaAccount placeholder)
npx prisma db seed

# Start in development mode
npm run start:dev
```

After starting, complete the Strava OAuth flow by navigating to:

```
http://localhost:3000/strava/auth
```

Authorize the app on Strava, then manually call the callback URL with the returned `code` parameter. Once connected, trigger an initial sync:

```bash
curl -X POST http://localhost:3000/strava/sync
```

## API Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/strava/auth` | Redirects to Strava OAuth authorization page |
| `GET` | `/strava/auth/callback` | Handles OAuth callback, exchanges code for tokens |
| `POST` | `/strava/sync` | Manually triggers an incremental activity sync |

The sync job also runs automatically every 6 hours via cron.

## Data Model

```
User
 └── StravaAccount       (OAuth tokens)
 └── Activity            (one per Strava run)
      ├── ActivitySplit   (1km splits from Strava)
      ├── ActivityLap     (detected or recorded laps)
      └── ActivitySecond  (second-by-second stream data)
```

Every entity carries a `userId` foreign key from day one, in preparation for multi-user support. Activities use `startDate` as the canonical correlation timestamp across future data domains (lifting, diet, etc.).
